### 2026-06-13 — Headroom compression-proxy routing on both executor paths

Two fork-only commits add optional routing of eligible agents' Anthropic traffic through a local **Headroom** compression proxy. Opt-in via `HEADROOM_PROXY_URL`; off by default.

- **`aaadaa9` — claude-executor (API-billed / `claude` provider).** The pay-as-you-go path, where token savings = real cost. `shouldRouteThroughHeadroom()` structurally hard-blocks evidence-critical modes (adversarial verifier, strict contract writer, precise) so their tool output / contracts always reach Anthropic uncompressed; only focused/reliable/objective/patient agents are eligible. Fail-safe: a configured-but-unreachable proxy is marked degraded for the run and the executor retries directly (the proxy attempt itself is no-retry) — a dead proxy never blocks or silently corrupts a workflow. Client construction is injectable (`clientFactory`) for network-free unit tests. 7 new tests; suite 141 green.
- **`a6d3f3a` — agent-sdk-executor (subscription / terminal-auth path).** The path most deployments actually use. Routes via the SDK's per-query `options.env`, setting `ANTHROPIC_BASE_URL` for that subprocess only; `process.env` is spread so the spawned Claude Code keeps its OAuth subscription credentials + PATH, while `ANTHROPIC_API_KEY` stays deleted (auth remains subscription-based). The evidence-critical safety gate is now a shared module `headroom-routing.ts` used by both executors, so verifier / contract-writer can never be routed regardless of provider. On subscription auth, compression saves usage-limit budget / context-window headroom / latency rather than per-token cost. 4 new tests; 145 green.

Consumed and A/B-tested by [[agent-stack]] the same day (see its 2026-06-13 entry): on a live subscription session efficacy was **~0.69%** — real claude-code / Agent SDK traffic uses prompt caching, which Headroom prefix-freezes rather than rewrites, so almost nothing compresses. The synthetic 66% doesn't translate. Recommendation: leave it off on the agent-sdk path. Routing correctness and the safety bypass were both confirmed under load.

### 2026-06-12 — Per-agent `max_turns` config with tool-aware defaults

`4e575aa` — agent blocks in `.aflow` now accept a `max_turns` field; the compiler carries it into `AgentDef` (`src/compiler.ts`, `src/types.ts`) and the Agent SDK executor (`src/executors/agent-sdk-executor.ts`) uses it as the turn budget. Without an explicit value, **tool-using agents now default to 50 turns** (the old hardcoded cap of 10 was killing implementer-style agents mid-task) and tool-less agents stay at 1. Covered by new tests in `tests/agent-sdk-executor.test.ts` + `tests/compiler.test.ts`.

This is the engine-side half of the fix; [[agent-stack]] consumed it the same day (`0cbdcfd`) by setting role-scaled `max_turns` budgets on every tool-using agent across all four `.aflow` workflows (implementer 80, writers/verifiers 40–50, etc.). Root cause surfaced on Rhythm issue #685: an implementer agent hit `error_max_turns` after 43s under the old 10-turn cap.

### 2026-06-11 — Fork patches: install claude-agent-sdk; resume accepts user_inputs at human gates

Two fork-only commits landed on `main` to unblock the [[agent-stack]] AgentFlow integration.

- **`2cd751b` — deps: install claude-agent-sdk; bump @anthropic-ai/sdk + zod to satisfy its peers.** The `agent-sdk` provider (subscription-credit auth, used by all tier aliases) needs `@anthropic-ai/claude-agent-sdk`, which upstream declares as an *optional* peer but never installs — so agent-sdk runs crashed at execution time even though `agentflow check` passed. Installing it pulled in its peers: `@anthropic-ai/sdk` `0.82 → 0.104` and `zod` `3 → 4`. The fork's own `src/` imports neither directly; `tsc` clean.
- **`9464e89` — resume: accept user_inputs for paused human_action_required phases.** `agentflow_resume` (MCP) and `agentflow resume` (CLI) previously took only `instance_id` + an approve flag, so the two human gates (`plan-and-issues` → `align`, `post-merge-smoke` → `manual_smoke`) could only be satisfied by hand-editing `<instance>.state.json` before resuming. Both surfaces now accept phase outputs directly:
  - MCP: `agentflow_resume({ instance_id, user_inputs: { align: { ... } } })`
  - CLI: `agentflow resume <file> --instance <id> --user-inputs '{...}'`
  - Threaded through `makeRunner` / `WorkflowRunner`'s existing `userInputs` option (the runtime already consumed it; only the entry points lacked a param).
- **Verified** end-to-end in `--mock` mode: `plan-and-issues` pauses at `align`, and resume with `--user-inputs` completes all four phases. This closes the state-file-edit workaround documented as known-gap 3 in the agent-stack 2026-06-10 AgentFlow milestone.
